# 本地音乐播放器开发路线图

## 路线原则

开发顺序以可运行、可验证、可扩展为优先级。

1. 先搭工程骨架，再做功能。
2. 先完成本地播放闭环，再扩展资料库体验。
3. 先保证 SQLite schema 和迁移可靠，再堆 UI 功能。
4. 先用 Electron/Chromium 原生媒体能力，后续再引入 FFmpeg。
5. 每一阶段都要有可运行应用、基础测试和 CI 检查。

## Phase 0：项目初始化

目标：建立可维护的开源工程基础。

主要任务：

1. 初始化 Electron + React + TypeScript + electron-vite 项目。
2. 配置 ESLint、Prettier、TypeScript strict。
3. 配置 Vitest。
4. 配置 Playwright 基础环境。
5. 配置 GitHub Actions PR 检查。
6. 建立 `src/main`、`src/preload`、`src/renderer`、`src/shared` 目录。
7. 建立 IPC contract 类型定义方式。
8. 配置 electron-builder 的基础打包。
9. 添加 README、LICENSE、贡献说明。

验收标准：

1. 本地能启动空白桌面应用。
2. React renderer 能显示基础壳界面。
3. `format`、`lint`、`typecheck`、`test`、`build` 可运行。
4. GitHub Actions 能在 PR 中跑通基础检查。

## Phase 1：播放器最小闭环

目标：完成选择本地音频文件并播放的最小体验。

主要任务：

1. 实现主窗口基础布局。
2. 实现底部播放栏。
3. 实现选择单个或多个音频文件。
4. 实现播放、暂停、上一首、下一首。
5. 实现进度条、seek、音量。
6. 实现播放队列内存模型。
7. 实现基础错误提示，例如格式不支持、文件不存在。
8. 保存最近一次音量和窗口尺寸。

验收标准：

1. 用户可以选择本地 MP3 文件并播放。
2. 播放队列可以切歌。
3. 进度和播放状态显示基本准确。
4. 重启应用后保留窗口尺寸和音量。

## Phase 2：SQLite 和本地资料库

目标：从单次打开文件升级为可持久化的本地音乐资料库。

主要任务：

1. 引入 SQLite 和 migration。
2. 建立 `tracks`、`scan_roots`、`schema_migrations` 等基础表。
3. 实现选择音乐目录。
4. 实现目录扫描。
5. 实现音频元数据读取。
6. 实现扫描结果批量入库。
7. 实现本地音乐列表。
8. 实现增量扫描：新增、修改、删除。
9. 记录扫描错误。

验收标准：

1. 用户可以添加音乐目录。
2. 应用能扫描目录并生成本地歌曲列表。
3. 重启应用后歌曲列表仍然存在。
4. 修改或删除本地文件后，重新扫描可以更新数据库。
5. 扫描大目录时 UI 不应明显卡死。

## Phase 3：歌单和搜索

目标：建立核心音乐管理能力。

主要任务：

1. 建立 `playlists` 和 `playlist_tracks`。
2. 实现创建、重命名、删除歌单。
3. 实现添加歌曲到歌单。
4. 实现歌单内排序。
5. 实现从歌单移除歌曲。
6. 实现歌曲、艺术家、专辑的基础搜索。
7. 实现最近播放。
8. 实现我喜欢。

验收标准：

1. 用户可以创建歌单并添加歌曲。
2. 歌单顺序重启后保持稳定。
3. 搜索可以找到歌曲、艺术家、专辑。
4. 最近播放和我喜欢可以正常记录和展示。

## Phase 4：界面完善

目标：接近正式播放器的日常可用体验。

主要任务：

1. 完善主界面视觉层级。
2. 实现歌曲表格排序和多选。
3. 实现播放队列抽屉。
4. 实现封面展示和封面缓存。
5. 实现设置页。
6. 实现主题能力，至少支持浅色和深色。
7. 实现空状态、加载状态、错误状态。
8. 优化长文本、省略、响应式布局。

验收标准：

1. 本地音乐、歌单、搜索、播放队列体验完整。
2. 大量歌曲列表滚动和操作稳定。
3. UI 在常见桌面窗口尺寸下不重叠、不溢出。
4. 基础无障碍属性和键盘操作可用。

## Phase 5：桌面系统集成

目标：先让应用在 macOS 上更像原生桌面播放器。

主要任务：

1. 系统托盘。
2. 全局媒体快捷键。
3. 应用菜单。
4. macOS 文件关联预研。
5. macOS dock 行为优化。
6. 自动更新预研。
7. 记录 Windows 和 Linux 后续适配清单。

验收标准：

1. 常见系统播放键可以控制播放。
2. 关闭窗口和退出应用行为符合平台习惯。
3. macOS 有可安装或可运行的构建产物。

## Phase 6：性能和搜索升级

目标：面向更大的本地曲库优化体验。

主要任务：

1. 歌曲列表虚拟滚动。
2. 扫描任务放入 worker thread 或 utility process。
3. 元数据解析并发限流。
4. SQLite FTS5 搜索。
5. 搜索结果排序权重。
6. 封面缩略图生成和缓存清理。
7. 播放进度更新节流。
8. 数据库 checkpoint 和维护策略。

验收标准：

1. 万级歌曲列表可流畅浏览。
2. 搜索响应稳定。
3. 大目录扫描不影响播放和界面操作。

## Phase 7：FFmpeg 扩展

目标：提升格式兼容性和媒体处理能力。

主要任务：

1. 确定 FFmpeg 分发方式和许可证边界。
2. 接入 ffprobe 读取更完整媒体信息。
3. 支持更多音频格式探测。
4. 为不支持的格式设计转码播放方案。
5. 生成波形数据。
6. 预研 ReplayGain 或响度分析。
7. 对 FFmpeg 子进程做超时、取消、异常处理。

验收标准：

1. 首期无法识别的常见无损格式能读取元数据。
2. 至少一种原先无法播放的格式可以通过转码链路播放。
3. FFmpeg 任务失败不会拖垮主进程。

## Phase 8：开源发布准备

目标：让项目适合公开协作和持续发布。

主要任务：

1. 完善 README。
2. 添加截图和功能说明。
3. 添加贡献指南。
4. 添加 issue 模板和 PR 模板。
5. 添加安全策略。
6. 添加 changelog。
7. GitHub Release 自动上传 macOS 产物。
8. 明确未签名构建的安装提示。

验收标准：

1. 新贡献者可以按 README 跑起项目。
2. Release 页面有 macOS 构建产物。
3. CI 状态清晰。
4. 基础文档覆盖项目定位、开发、测试、打包。

## 首期 MVP 范围建议

建议第一个可发布版本只包含：

1. 本地目录扫描。
2. 歌曲列表。
3. 基础播放控制。
4. 播放队列。
5. 歌单。
6. 搜索。
7. 最近播放。
8. 我喜欢。
9. 浅色和深色主题。
10. macOS 基础构建。

暂缓：

1. FFmpeg。
2. 在线歌词。
3. 云同步。
4. 自动更新。
5. 复杂音效。
6. 账号系统。
7. 插件系统。

## 风险清单

### Native module 构建风险

`better-sqlite3` 是 native module，需要适配 Electron ABI。当前必须先验证 macOS 安装、rebuild 和运行；后续适配 Windows/Linux 时再补充分平台验证。

应对：

1. 固定 Electron 和 better-sqlite3 版本。
2. CI 先做 macOS 构建。
3. 升级 Electron 时单独做 native module 验证。

### 媒体格式支持风险

不引入 FFmpeg 时，格式支持取决于 Chromium/Electron 和平台能力。

应对：

1. README 明确首期支持范围。
2. 对不支持文件展示友好错误。
3. 保留 FFmpeg 扩展接口。

### 大曲库性能风险

歌曲数量上来后，扫描、搜索、封面、列表渲染都可能成为瓶颈。

应对：

1. 扫描和元数据解析任务后台化。
2. 数据库写入使用事务。
3. 列表使用分页或虚拟滚动。
4. 搜索迁移到 FTS5。

### UI 复杂度风险

播放器 UI 状态很多，播放状态、队列状态、资料库状态容易耦合。

应对：

1. 播放队列独立建模。
2. 资料库和播放状态分离。
3. React 状态按 feature 划分。
4. IPC contract 类型集中管理。

### 后续跨平台差异风险

Windows、macOS、Linux 在路径、权限、菜单、托盘、安装包上差异明显。当前阶段只处理 macOS，不能把 macOS 假设写死到业务模型里。

应对：

1. 平台差异集中放在 `src/main/system`。
2. 不把路径规则写死。
3. Windows/Linux 适配前补充分平台 CI 构建。

## 推荐近期任务顺序

1. 确认包管理器：npm 或 pnpm。
2. 初始化 Electron + React + TypeScript + electron-vite。
3. 建立基础 CI。
4. 实现空白窗口和应用布局。
5. 定义 IPC contract。
6. 实现选择文件并播放。
7. 引入 SQLite migration。
8. 实现本地目录扫描。
9. 实现歌曲列表和播放队列。
10. 实现歌单和搜索。
